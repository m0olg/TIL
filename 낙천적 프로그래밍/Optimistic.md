# 02 비동기 vs 낙천적 프로그래밍
- 비동기 : 데이터 처리를 기다리는 백엔드 / 로직의 영역으로 데이터가 올 때까지 화면을 멈추거나 기다림
- 낙천적 프로그래밍 : 사용자에게 즉각적인 반응을 보여주는 UI/UX의 영역

### 특징
- 先 UI 업데이트, 後 서버 통신 : 데이터가 서버에 저장되기도 전에 일단 성공한 것처럼 UI를 먼저 바꾸어버림
- 대기 시간 최소화 : 카카오 맵처럼 세부 정보 창을 띄울 때, 데이터가 다 올 때까지 사용자를 기다리게 하지 않고 틀(UI) 먼저 보여줌

### 차이점
- 비동기 : 작업할 데이터가 없거나 에러가 나면 "실패"로 간주하고 에러 처리
- 낙천적 프로그래밍 : 작업할 게 없으면 일단 nil(데이터 ❌) 상태로 뷰를 비워두거나 스켈레톤을 보여주며 놔둠 \
즉, 에러로 끊어버리는게 아닌 "언젠가 채워질 것"으로 보고 유지

#### 좋아요 버튼 예시 ⬇️⬇️
> ### 1) 비동기 방식 (Wait for Server)
- 사용자가 좋아요 클릭 → 서버에 API 호출 (Request) → 서버 응답 (Respones) 올 때까지 대기 → 응답 성공 시 하트 색깔 변경
- 네트워크가 느리면 하트가 한참 뒤에 빨갛게 변함
- 실패하면 시트가 뜨거나 에러 팝업이 뜸

> ### 2) 낙천적 프로그래밍 방식 (Optimistic UI)
- 사용자가 좋아요 클릭 → 즉시 하트를 빨갛게 변경 (Local Update) → 동시에 서버에 API 호출
- __성공 시__ : 그대로 유지 (사용자는 매우 빠르다고 느낌 !!)
- __실패 시__ : 슬그머니 다시 하트 색을 원래대로 되돌림 (Rollback) \
Rollback - 작업 실패 시, 이전 정상 상태로 되돌리는 것
- 서버 저장과 상관없이 일단 내 폰(Local) UI 안에서만 먼저 작동시키고 이전 상태를 유지하는 방식 ‼️
