## 0-2) mvc
- ### MVC의 세 가지 역할
    - model
        - 앱이 무엇을 하는지, 어떤 데이터를 다루는지 정의
        - 서버 통신 결과인 JSON 데이터를 담은 구조체(Struct)나, 데이터를 처리하는 로직이 여기에 해당
        - 특징 : 화면에 무엇이 그려지는지 전혀 모르고 오직 데이터만 신경 씀
    - View (사용자 인터페이스)
        - 사용자에게 보이는 화면 그 자체임 \
        (`UIButton`, `UILabel` 같은 것들)
        - 특징 : 데이터를 직접 수정 하지 ❌, 받은 데이터를 보여주기만 함 사용자가 버튼을 누르면 눌렀다고 컨트롤러에게 알려줌
    - Controller (중간 관리자)
        - model과 view 사이를 연결하는 다리 역할
        - ios에선 보통 'UIViewController'가 이 역할을 수행함 !!
        - 동작 방식은 사용자가 View에서 버튼을 누르면, Controller가 데이터가 필요하네하고 model에게 서버 통신을 시킴 \
        그 결과가 돌아오면 다시 View에게 "이거 화면에 그려줘"라고 명령

- ### 서버 통신에서의 흐름 (mvc 관점에서)
    1. View : 사용자가 '새로고침' 버튼을 누름
    2. Controller : 버튼 액션 함수에서 서버 통신 코드 실행
    3. model : 서버에서 받아온 JSON 데이터를 구조체에 저장
    4. Controller : 모델 데이터가 준비 되었으니, View에 데이터 전달
    5. View : 화면에 사용자 이름과 프로필 사진이 나타남

# 01 View, model, VM
## 1) View
view = 퍼블리싱 + 네트워크 (상태 반영)
- 퍼블리싱은 디자인을 코드로 옮기는 것
- 화면이 어떻게 보일지(UI)뿐만 아니라 데이터가 바뀌었을 때 어떻게 변할지도 결정 \
그래서 우리의 생각과는 다르게 _젤 어려움_

# 2) model
그나마 세개 중에 젤 쉬운 것, 데이터의 원재료 (가장 단순한 형태)
### 요청과 응답
- 요청 (Request) : 클라이언트가 서버에게 "데이터 줘ㅠㅠ!!"라고 말함
- 응답 (Respone) : 서버가 클라이언트에게 데이터를 보내줌
이 두개는 서버와 클라이언트 간 연결을 할 때 사용함 \
_서버에서 클라이언트_로 갈 땐 __응답__, _클라이언트에서 서버_로 갈 땐 __요청__

#### 서버엔 DB가 있음
- 서버의 데이터 JSON → Swift 객체 (model)
- 이 과정에서 디코딩을 사용함 \
_디코딩이란? 서버가 보내준 데이터(주로 JSON 문자열)를 Swift가 이해할 수 있는 클래스나 구조체 형식으로 변환하는 것 즉, 언어를 바꿔주는 방식_

### 예시 ⬇️⬇️
> 만약 게시물이라는 앱이 있다고 치고 \
Post라는 서버의 DB에는 제목, 내용, 날짜 등이 있어야함 \
(제목 : 닭발, 내용 : 직화닭발, 날짜 : 5.6) 이런 형식으로 DB에 여러개가 있을거임 \
\
이제 클라이언트에서 DB에 있는 내용을 보여주어야 사용자가 볼 수 있음 (저장된 값을 받고 사용자에게 보여지게 해야지 사용자가 볼 수 있음) \
⬇️\
그리고 서버에서 클라이언트에게 응답을 받으면 DB에 있는 데이터들을 받아온 다음 \
사용자에게 보여줌 (이 과정에서 디코딩을 통해 서버에 있는 언어를 Swift언어로 바꾸어줌)


\
\
서버에서 스위프트로 디코딩을 할 때 JSON(제이슨)이라는 언어가 된 다음 스유가 됨 \
__= 서버 → JSON → Swift__



#### 제이슨의 구성요소
- `key : value` (ex. `"title" : "닭발"`) \
`보내는 값 : 자료형`

즉 서버에 있는 값을 key랑 value로 바꾸고 난 다음 Swift가 됨

### JSON에서 Swift로
(아까 쓴 거에서 이어서)
- value가 Swift의 자료형(타입)으로 변환
- 제목 : "닭발" (String)
- 가격 : 15000 (Int)
- 날짜 : "2024-05-10" (String으로 받아서 나중에 Date로 변환 )

##### Q. 그러면 왜 그대로 안 보내주고 자료형에 맞춰서 받을까?
##### A. Swift는 타입에 매우 엄격하고 타입 추론이 가능함
- 타입 추론 : Swift는 타입추론이 가능함 (알아서 자료형 판단) 그래서 코드 짤 때 하나씩 안 써도 됨
- 하지만, 서버에서 "닭발(String)을 보냈는데, 클라이언트 모델에서 실수로 숫자(Int)로 선언해두면 Swift가 오류를 내뿜음
- 그래서 JSON에서 나누는 이유는 클라리언트, 서버에 각각 같은 타입이 있어야 하기 때문
- __즉, 모델에서 정한 '타입'이라는 요구 사헝이 서버와 일치해야지만 데이터가 안전하게 전달됨__

    ---
    ---

- model이라는 것에서 하는 일은 \
데이터의 형식만 정해줌, 그래서 우리가 생각하는 닭발의 값을 보내주는 코딩은 다른 파일에서 함

- model에서 필요한 요구 사항이 다 정해짐 \
그 요구 사항 중 하나가 타입이 같아야한다는 것이고 서버에서 요구하는 방식은 닭발인데 클라이언트에서 이진법으로 받아버리면 \
숫자로 출력이 되므로 값으로 바꾸어주어야함 \
그래서 똑같이 클라이언트에서도 string으로 바꿈

## 2-1) DTO (데이터 전송)
- __데이터 전송용 모델__
- 서버의 Key값 (ex. `title_A`)을 그대로 받아내는 '바구니'

그래서 이걸 유연하게 써주기 위해 바꾸어 줘야함 (Domain Model를 통해!)
## 2-2) Domain model
- __앱 내부에서 실제로 사용하는 모델__
- `title_A`를 `title`로 바꾸는 등 우리 앱에 맞게 유연하게 변경
- 이렇게 DTO를 Domain Model로 변환해주는 걸 **맵핑**이라고 함

- 모델을 바탕으로 실제 데이터("닭발")를 가져와서 뷰가 보여주기 좋게 가공
- 서버에 데이터를 요청하고 응답받는 로직을 실제로 코딩하는 곳이 바로 여기
