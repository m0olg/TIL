# 01 서버통신이란?
→ 내 폰(앱)이 필요한 정보를 얻기 위해 멀리 있는 서버에 말을 걸고 대답을 듣는 즉, 데이터를 주고 받는 과정으로 인스타 새로고침 할 때, 카카오톡 할 때도 서버통신이 일어남

### 특징
- 서버는 눈에 보이지 않음
- 클라이언트와 통신함
- 비동기 처리 많이 사용
- 데이터 처리 중심

#### Q. 서버통신 없이도 채팅을 구현할 수 있을까?
A. 서버 통신을 안해도 사용자 입장에서 채팅처럼 보이게 디스플레이를 구현할 수는 있지만 사실상 '빈껍데기'임 \
    = 서버통신을 안해도 사용자 입장에서 채팅처럼 보이게 디스플레이를 구현 가능



\
\
__1차원적으로 보면 모든게 데이터임 !!__ \
ex.) 사람으로 보면 눈, 코, 입, 손, 발, 팔...

# 03 서버
→ 가공한 데이터(DB)로 뭔가를 할 수 있게 해주는게 서버임

ex.) 맥도날드 \
    빅맥이라는 메뉴를 시키고 이걸 주방으로 보냄 \
    주방에서 메뉴를 만들고 다시 카운터로 보냄

= 즉, 주방이 서버고 빅맥을 받는 카운터가 클라이언트(UI), 빅맥인 데이터이고 만드는 레시피는 DB의 기준임


# 04 동기
### 4-1 동기란?
→ 서버에 응답을 무조건 받아야지만 다음 작업을 할 수 있는 것 (ex. 로그인, 회원가입 등) \
**[동기는 직렬 구조 ⭐️⭐️]**

### 4-2 동기의 특징
- 순차적으로 실행됨
- 기다리는 동안 아무것도 못함 (비효율적)
- 간단 계산이나 빠르게 끝나는 작업, 순서가 중요한 로직에 주로 사용


# 05 비동기
### 5-1 비동기란?
→ 먼저 뷰가 생성되고 그 다음에 통신도 같이 돌아가고 동시에 작업 가능 (병렬구조기 떄문) \
**[비동기는 병렬 구조 ⭐️⭐️]**
### 5-2 비동기의 특징
- 작업 요청만 하고 결과는 나중에 받음 (비순차적 실행)
- 시간 낭비 없고 대기 시간도 활용함
- 파일 입출력, DB 작업, 이미지 다운에 주로 사용

# 07 리스폰 필드 (응답 필드)
- 서버에서 필요한 데이터들을 요구 \
ex.) 유저를 구성하기 위해서는 (회원가입을 하기 위해서는) 이름, 비밀번호, 메일 등이 필요함
    - 각각의 타입에 맞게 받아야함 (이름-문자열, 비밀번호-문자, 숫자)

    ### 에러로깅
    - 서버에서 보내주는 에러 값

    ### API
    - 로그인이라는 정보를 담은 작은 서버가 로그인 API \
    (기능 서버 ‼️)

# 08 프로젝트 도중 발생할 수 있는 갈등
### ① 서버 ↔ 디자인
- 원인
    - 디자인은 UI/UX의 완성도 중요 \
    - 서버는 데이터 구조, 성능, API의 구조가 중요하므로
- 갈등 포인트
    - 데이터 구조 / UI 요구사항
    - 실시간 처리 / 서버 부하
### ② 디자인 ↔ 클라이언트
- 원인
    - 클라이언트는 추상적인 요구
    - 디자인은 구체적인 결과물 필요
- 갈등 포인트
    - 기능명세서 누락
    - 기대치 불일치
### ③ 서버 ↔ 클라이언트
- 원인
    - 클라이언트는 기능 중심
    - 하지만 서버는 기술적 제한 존재
- 갈등 포인트
    - 성능 / 요구 사항
    - 개빌 기간 / 기능 범위
